## Известная проблема — НЕ регрессия этой задачи

`POST /clients` возвращает `500` при попытке провижининга VLESS: `add xui client: xui: unexpected status 404`.

**Причина установлена**: работающая панель `ghcr.io/mhsanaei/3x-ui:latest` (образ подтягивается как `:latest`, контейнер поднят 2 недели назад) обновилась и **сменила API**. Прямая проверка `curl` в обход Go-кода подтверждает: `POST /panel/api/inbounds/addClient` → `404` на голом HTTP-уровне, при этом `GET /panel/api/inbounds/list` → `200` работает штатно. Извлечение строк из бинаря панели (`strings /app/x-ui`) показало, что маршруты клиентов переехали с `/panel/api/inbounds/{addClient,updateClient,delClient}` на **новый top-level ресурс** `/panel/api/clients/{add,update/,del/,get/,list,traffic/,...}` (подтверждено: `GET /panel/api/clients/list?inboundId=1` → `200`).

Это расхождение существовало бы независимо от рефакторинга EdgeProvisioner в этой задаче — `internal/xui/client.go` (не тронутый по существу, только обёрнут интерфейсом) слал бы точно такой же 404 и до миграции. Не входит в объём ТЗ P2.1 (явно: «это чистый рефактор без изменения поведения», раздел 4.3 ТЗ) — исправление `internal/xui` под новый контракт `/panel/api/clients/*` требует отдельной задачи (новые формы запроса/ответа нужно сверить с реальным поведением панели, не только с путём).

Регрессия по функционалу, который поддаётся проверке без живой панели, подтверждена: юнит-тесты `internal/clients` (мок 3x-ui через `httptest`) зелёные — включая rollback-путь при сбое `AddUser`/`syncHysteriaUsers`, что и есть тот код, который в реальности упадёт на живом 404 точно так же, как упал бы до рефакторинга.
