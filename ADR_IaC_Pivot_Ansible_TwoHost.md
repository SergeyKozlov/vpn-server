# ADR — Переход на IaC (Ansible), разделение control-plane/data-plane между хостами А и Б

**Дата:** июль 2026 | **Статус:** Решения зафиксированы, реализация не начата | **Продолжение:** ADR-1–9 из `VPN_Project_Status_and_ADR.md`

---

## Контекст и мотивация

До этого момента вся разработка велась напрямую на боевом сервере (`92.60.75.196`, хост А): Claude Code на сервере получал ТЗ, менял конфиги/код на живой машине, коммитил. Это сработало для MVP, но уже дважды создало проблемы, которые IaC-подход снимает системно:

1. **Тихие расхождения без единого источника истины.** Диагностика Hysteria2 показала: живой конфиг, `HYSTERIA_CONFIG_PATH` из `.env` и compose-сервис `hysteria2` — три разных файла с разным содержимым, накопившихся за счёт ручных правок в разное время. Без декларативного описания состояния такие расхождения невозможно заметить, пока что-то не сломается.
2. **Тестирование на проде.** Каждая диагностика/фикс выполнялись на боевой машине, с ручными предосторожностями (проверка активных сессий, ручной откат). Это работает, но не масштабируется и не профессионально как практика.

**Решение:** переход на Ansible-driven IaC с разделением на два хоста — при этом сохраняя уже принятые ранее рабочие принципы (control points, отчёты в `.md`, git commit+push по подтверждению, различие «точечный фикс / крупное решение» в объёме согласования).

---

## Решения

### ADR-10: Топология — хост А (control + data plane) и хост Б (app plane, эталон)

- **Хост А:** IaC-код (Ansible), тестовые/черновые конфиги, **и PostgreSQL как боевой сервис** (см. ADR-13 — уточнение относительно исходного плана).
- **Хост Б:** эталонный прод — все stateless-сервисы приложения (`api`, Caddy, 3X-UI, Hysteria2), полностью пересобираемые Ansible'ом с нуля. Доменные имена переключаются на Б.
- Оба хоста — тот же провайдер, идентичная конфигурация (кроме IP), сейчас 1 vCPU / 1 ГБ RAM (см. ADR-12 про решение оставить как есть).

### ADR-11: Хост Б собирается «начисто» по Architecture Concept, не зеркалирует накопленный долг хоста А

- Провижининг клиентов на Б с самого начала строится поверх AC-DB схемы (`user_credentials`, `nodes`, `node_protocols`), а не поверх `legacy_clients_phase1` (P1-таблица, которую сейчас использует `/clients` на А).
- Это прямо связано с открытым вопросом «`legacy_clients_phase1` vs AC-DB» из предыдущей диагностики — IaC-переезд используется как естественная точка исправления, а не как повод отложить ещё раз.
- **Компромисс:** нельзя просто скопировать состояние хоста А Ansible'ом — нужно осознанно спроектировать, что именно разворачивается на Б (миграции с нуля по актуальной схеме, а не дамп текущей БД приложения; конфиги — по AC, а не по факту с А).

### ADR-12: RAM хоста Б остаётся 1 ГБ; риск компенсируется сборкой артефактов вне хоста

- Наблюдение из хардненинга (`free -h` → 77 МБ свободно при работающем стеке) показывает, что запас минимальный. Решено не увеличивать тариф сейчас.
- **Митигирующий IaC-принцип:** Go-бинарники и Docker-образы **не собираются на целевом хосте**. Сборка — на хосте А (или локально/CI), артефакт доставляется на Б Ansible-задачей (`copy`/`fetch` либо через приватный registry). Это устраняет именно тот сценарий (параллельная компиляция + работающий стек), который создал риск OOM при хардненинге.
- **Открытый риск:** даже без сборки, одновременная работа Postgres (если он всё же где-то рядом)/3X-UI (embed Xray-core)/Hysteria2/Caddy/api на 1 ГБ остаётся тесной. Не блокирует старт, но держать в уме при появлении новых сервисов.

### ADR-13: PostgreSQL — на хосте А, приложения — на хосте Б; доступ через WireGuard-туннель

- **Решение:** отступление от исходного тезиса «хост А — только тестовые конфиги и IaC» (см. изначальный план, пп. 6/8) — Postgres размещается на А как боевой сервис. Причина — приоритет на портируемость: при смене провайдера копируется БД + всё, что не покрыто IaC, а не переносится вся стек-конфигурация целиком.
- **Обязательное условие:** у провайдера нет приватной сети/VLAN между VM (подтверждено) → доступ Б→А к Postgres **только через WireGuard-туннель** между хостами. Публичный `5432` без туннеля — недопустимо.
- WireGuard-роль в Ansible — предварительный шаг перед Postgres-ролью и перед любой app-ролью, которая обращается к БД по сети.
- `pg_hba.conf`/firewall на А — allowlist только по WireGuard-интерфейсу/подсети туннеля, не по публичному IP хоста Б.

### ADR-14: Провижининг VM — ручной; Ansible только конфигурирует ОС и сервисы

- Terraform/аналоги для создания самих VM у провайдера — не используются. Хосты создаются вручную (как сейчас), Ansible берёт на себя всё после появления SSH-доступа к чистой машине.
- **Компромисс:** воспроизводимость ограничена уровнем «ОС + сервисы»; сам факт существования VM определённой конфигурации у конкретного провайдера — вне кода, в голове/памятке.

### ADR-15: Принцип портируемости — IaC покрывает воспроизводимое, состояние переносится вручную

- Явная цель (сформулирована владельцем проекта): в любой момент можно сменить провайдера. Тогда: 1) Ansible поднимает новую пару хостов А'/Б' с нуля из репозитория; 2) БД (дамп/`pg_dump`+`restore`) и всё, что физически не описано в IaC (TLS-сертификаты, если не re-issued автоматически через Cloudflare DNS-01, секреты) — копируются отдельно, вручную/скриптом миграции.
- Это разделяет ответственность чётко: **Ansible = cattle (пересобираемая инфраструктура и конфиги), ручной перенос = pet (данные и то, что нельзя вывести из кода)**.
- Из этого следует требование к репозиторию: явный список того, что **не** в IaC (в первую очередь: содержимое БД, секреты — держать в `ansible-vault` или внешнем секрет-менеджере, не в открытом виде в репо).

### ADR-16: Порядок фаз IaC-переезда (0–7) — зафиксирован

Ранее (см. п.4 «Что осталось нерешённым» ниже) это был предварительный набросок, не финализированный как ТЗ. Зафиксирован как принятое решение при ревью спек `phase-2-wireguard` (`vpn-iac`), 2026-07-24:

| # | Фаза | Содержание |
|---|---|---|
| 0 | Подготовка | Bootstrap `vpn-iac` (tooling, inventory, connectivity к обоим хостам) |
| 1 | Baseline | `roles/common` — deploy-user, ssh-hardening, ufw, fail2ban, control-node toolchain (на обоих хостах) |
| 2 | WireGuard | Туннель А↔Б, ключи, фаервол по `wg0`, сетевая достижимость Postgres через туннель (ADR-13) |
| 3 | Postgres | Postgres как декларативный Ansible-сервис на А; `pg_hba.conf`, роли/пользователи БД (вне рамок Фазы 2) |
| 4 | Сборка артефактов вне хоста | Go-бинарники/Docker-образы собираются не на целевом хосте (ADR-12), доставляются Ansible'ом |
| 5 | App-tier на Б | `api`, Caddy, 3X-UI, Hysteria2 — эталонная пересборка на хосте Б под управлением Ansible (ADR-10, ADR-11) |
| 6 | Проверка | End-to-end проверка стека на Б перед переключением трафика |
| 7 | Cutover доменов | Переключение DNS на Б; судьба хоста А после переключения — отдельный вопрос (см. п.6 «Что осталось нерешённым») |

Порядок ролей внутри фаз 2–3 (WireGuard строго до Postgres и до любой сетевой app-роли) — уже зафиксирован в ADR-13 и этим не меняется.

## Что осталось нерешённым (следующие шаги после этого ADR)

1. **Структура Ansible-репозитория** (роли, inventory, каталоги) — следующий деливеробл после этого документа.
2. **Управление секретами** — `ansible-vault` внутри репо или внешнее хранилище (не обсуждали конкретно).
3. **Backup-стратегия для Postgres на хосте А** — не обсуждали (WAL/pg_dump расписание, куда бэкапы кладутся).
4. ~~Порядок ролей и точный acceptance для первого прогона~~ — **решено**, см. ADR-16 (порядок фаз 0–7 зафиксирован). Точный acceptance по-прежнему определяется по-фазово (для Фазы 2 — см. `vpn-iac/openspec/changes/phase-2-wireguard/`).
5. **Судьба хоста А как «эталона тестового окружения»** — раз он теперь несёт боевой Postgres, нужно отдельно решить, что на нём считается «тестовым», а что — боевым (сейчас там же `.env.test`/`vpn_test` БД по существующей практике — держать физически на одной машине с боевой БД или разносить дальше?).
6. **Cutover-план для доменов** (п.9 исходного списка) — когда и как переключать DNS на Б, что происходит с А в момент переключения (полностью выключается, остаётся как staging?).

## Что НЕ меняется из уже принятых правил проекта

- Control points и разделение «точечный фикс / крупное решение» по объёму согласования — сохраняется.
- Отчёты сохраняются в `.md` и коммитятся; `git push` — только по явному подтверждению.
- `Architecture_Concept_VPN_Project.md` остаётся источником истины для бизнес-логики/схемы; этот ADR — источник истины для инфраструктурной топологии поверх него.
