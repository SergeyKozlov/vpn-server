# Отчёт — миграция `internal/xui` на API 3X-UI v3.5.0 (`/panel/api/clients/*`)

## Контекст и причина задачи

`ai_from_local/Phase1_Audit_Report.md` (раздел 7) и `ai_from_local/P2.1_Schema_Redesign_Report.md` (раздел «Известная проблема») зафиксировали живую регрессию, обнаруженную независимо от рефакторинга P2.1: `POST /clients` возвращал `500` с ошибкой `add xui client: xui: unexpected status 404`.

Причина: контейнер `ghcr.io/mhsanaei/3x-ui:latest` обновился до v3.5.0, где клиенты стали самостоятельными сущностями (`ClientController`, таблица `client_records`), а старые роуты `/panel/api/inbounds/{addClient,updateClient,delClient}` были полностью удалены (не задеприкейчены — `web/web.go` отдаёт `404` через `engine.NoRoute`). Справочный документ `ai_from_local/3X-UI_v3.5.0_Client_Management_API_Go_Client_Migration_Reference.md` (подготовлен заранее, до этой задачи) дал первоначальные гипотезы по новому контракту, но с явной оговоркой — «подтвердить точные пути и схему на `/panel/api-docs` своей панели перед тем, как фиксировать Go-структуры».

## Что сделано

### 1. Верификация контракта напрямую против живой панели (не только по справочному документу)

`{webBasePath}/panel/api-docs` вернул `307` (редирект на `/`, доступен только по cookie-сессии, не по Bearer), поэтому спецификация снималась прямыми `curl`-пробами против реального инстанса (используя `.env`'s `XUI_BASE_URL`/`XUI_API_TOKEN`). Это выявило два расхождения с гипотезами справочного документа:

- **Ключ поиска — email, не UUID и не числовой `id`.** `GET /panel/api/clients/get/<numeric-id>` и `.../get/<uuid>` — оба `404 record not found`; только `GET /panel/api/clients/get/<email>` работает. То же для `update/:id`, `del/:id`, `traffic/:id` — везде `:id` фактически email.
- **Поле VLESS-credential в теле `add` — `"id"`, не `"uuid"`.** Проба с полем `"uuid"` в теле создания клиента была **молча проигнорирована** — панель сгенерировала свой UUID вместо переданного, без какой-либо ошибки. Проба с полем `"id"` — сработала, сохранённый UUID совпал с переданным. Если бы это не было проверено вживую и код отправлял `"uuid"`, каждый сгенерированный и зашифрованный в БД VLESS UUID был бы тихо заменён панелью на другой — клиент получил бы нерабочий конфиг без единой ошибки в логах.
- Форма тела `add`: `{"client": {...}, "inboundIds": [N]}` (не плоский объект и не `{"clients": [...]}` — обе эти формы были опробованы и дали ошибки валидации, которые и навели на правильную форму).
- `update/:email` — тело плоское (без обёртки `"client"`), и это **полная перезапись записи**, не partial-patch: поле `enable`, не переданное явно, было обнулено в `false` в тестовом прогоне. Это задокументировано как doc-комментарий в коде (`internal/xui/client.go`), чтобы будущий код не занёс частичный объект.

Все пробные записи (`probe-test`, `probe-test2`, `probe-test3`) созданы и удалены в рамках сессии, не оставлены в панели.

### 2. Код

- `api/internal/xui/types.go`: `Client` — убран `clientSettings`/старый `addClientRequest` (обёртка settings-как-строка), добавлен `addClientRequest{Client, InboundIDs}`; `TgID` сменил тип `string` → `int64` (справочный документ отдельно предупреждал: `tgId` должен быть числом, пустая строка вызывает падение при последующем редактировании клиента на панелях, мигрировавших с 2.x).
- `api/internal/xui/client.go`: `AddClient` → `POST /panel/api/clients/add`; `UpdateClient(ctx, email, client)` → `POST /panel/api/clients/update/{email}` (сигнатура потеряла параметр `inboundID` — привязка к инбаунду в update не участвует); `DeleteClient(ctx, email)` → `POST /panel/api/clients/del/{email}` (сигнатура потеряла `inboundID`); `GetClientTraffics(ctx, email)` → `GET /panel/api/clients/traffic/{email}`; `GetClientTrafficsByID` удалён (не было продакшен-вызывающего кода вне тестов, старый роут `getClientTrafficsById` не существует в v3.5.0).
- `api/internal/provisioner/threexui.go`: `RemoveUser`/`RotateCredential`/`GetTraffic` переведены на использование `email` вместо VLESS UUID как ключа поиска — это согласуется с уже существующей семантикой интерфейса (`Hysteria2Provisioner` использует `username` как такой же человекочитаемый ключ), просто раньше не было явно сформулировано для xui-адаптера. Уточнён doc-комментарий `EdgeProvisioner` в `provisioner.go` про это соглашение.
- `api/internal/clients/service.go`: исправлен смежный баг в пути отката — при неудаче синхронизации Hysteria2 откат вызывал `s.vless.RemoveUser(ctx, "vless_reality", vlessUUID)` (передавая UUID), что с новым API привело бы к тихому непрохождению удаления orphaned-клиента на панели (poиск по UUID всегда `404`). Заменено на `email`.

### 3. Тесты и верификация

- Обновлены `api/internal/xui/client_test.go` (пути, форма тела, добавлен `TestUpdateClientSendsFlatClientBody` — фиксирует именно full-replace семантику `update`) и `api/internal/clients/service_test.go` (пути моков `/panel/api/clients/add`, `/panel/api/clients/del/*` вместо старых `addClient`/`delClient`).
- `go build ./...`, `go vet ./...` — чисто.
- `go test ./...`, включая реальные Postgres-интеграционные тесты (`DATABASE_URL` из `.env`, база `vpn-postgres`) — все пакеты зелёные.
- **Живой end-to-end смоук-тест** против реальной панели (не только httptest-мока): поднят `go run ./cmd/api` на `:18080` с реальными `.env`-переменными, создан тестовый админ через `cmd/createadmin`, `POST /login` → `204` + валидная cookie, `POST /clients` с пустым телом (все дефолты) → **`201 Created`** с полным сгенерированным клиентом. Прямой `curl` на панель (`GET /panel/api/clients/get/<email>`) подтвердил, что запись реально создана и `uuid` совпадает 1-в-1 с тем, что вернул наш API. Все тестовые артефакты подчищены: xui-клиент удалён на панели, строка `legacy_clients_phase1` удалена из БД, тестовая запись убрана из `hysteria/config.yaml` (была дозаписана туда синком, реальный `hysteria-server.service` не перезагружался — `HYSTERIA_RELOAD_COMMAND` в смоук-тесте был подменён на `true`, поэтому живой сервис не затронут), тестовый админ `smoketest` удалён, фоновый процесс `go run` остановлен.

## Итог

Регрессия из `Phase1_Audit_Report.md` §7/§10.1 (`POST /clients` 500 на реальной панели) закрыта. Изменение закоммичено (`09fc1d9`, ветка `master`, запушено в `origin/master`) и не задевает БД/миграции — только клиент 3x-ui и его непосредственных потребителей (`clients.Service`, `ThreeXUIProvisioner`).

## Не входит в объём этой задачи (замечено попутно, не исправлялось)

- Публичного `revoke`/`DELETE /clients/{id}` эндпоинта в HTTP API всё ещё нет (AC-6.4, см. `Phase1_Audit_Report.md`) — `xui.DeleteClient`/`ThreeXUIProvisioner.RemoveUser` вызываются только как внутренний rollback при сбое `Create`, не как эксплуатационный путь.
- Вторичные роуты нового API (`bulkCreate`, `attach/detach`, `onlines`, `list/paged`) не реализованы — не требуются ни одним существующим потоком.
