# Architecture Concept — VPN-Project

> **Назначение:** источник правды по архитектуре сервиса (control-plane, data-plane, схема БД, API, клиентская логика подключения). На основе этого документа в чате «Сервер и инфраструктура» будут генерироваться ТЗ для Claude Code (консоль сервера) и Claude Code в VS Code (генерация Ansible/Docker/конфигов/SQL).
>
> **Область:** всё, кроме программирования конкретных модулей и кроме деталей биллинга (биллинг — отдельный чат, здесь только заглушки).
>
> **Статус полей:** решения помечены как `[УТВ]` (утверждено в обсуждении), `[ДЕФОЛТ]` (разумное значение по умолчанию, можно пересмотреть), `[ОТЛОЖЕНО→<чат>]` (проработка в другом чате).
>
> **Дата:** 19.07.2026. Каждый раздел имеет ID (напр. `AC-DB-3`) для ссылок из будущих ТЗ.

## AC-0. Ключевые принципы (сводка утверждённого)

- **AC-0.1.** Разделение **control-plane** (управление: сайт, API, БД, биллинг, аккаунты, выдача параметров) и **data-plane** (туннель: edge-узлы, протоколы). Разные стратегии устойчивости: control-plane — редундантность (несколько доменов, fallback); data-plane — мимикрия (ротация fingerprint, decoy-SNI, несколько протоколов). `[УТВ]`
- **AC-0.2.** Адаптивность — **функция клиента**, не сервера. Сервер поднимает несколько транспортов одновременно; клиент выбирает/переключает протокол по условиям сети. `[УТВ]`
- **AC-0.3.** VLESS Reality — основной протокол (TCP/443). Hysteria2 — резервный (UDP/QUIC). `[УТВ]`
- **AC-0.4.** Источник истины — **БД control-plane**. Состояние edge-узлов (3X-UI и пр.) — производное, приводится в соответствие reconciler'ом. `[УТВ]`
- **AC-0.5.** Масштаб первого этапа: 1 регион (Казахстан), 1 edge-узел, до ~10 узлов вручную. Схема данных — сразу многоузловая. `[УТВ]`

## AC-1. Инфраструктура и DNS

- **AC-1.1.** Домен этапа тестирования — единый: `videcdn.net`. Боевые домены закупаются после тестов и разносятся на разные SLD. `[УТВ]`
- **AC-1.2.** `www.videcdn.net` → проксирование Cloudflare + геоблокировка РФ (запрет доступа из РФ на уровне Cloudflare Firewall Rules). `[УТВ]`
- **AC-1.3.** `api.videcdn.net` → **только DNS** Cloudflare (без проксирования) → голый IP edge/control-плейн-сервера `92.60.75.196`. Причина: Cloudflare в РФ подвергается прицельному DPI-вмешательству (по ECH), критичный для клиентов API не должен зависеть от инфраструктуры Cloudflare. `[УТВ]`
- **AC-1.4.** **Риск (принят на этап теста):** блокировка SLD `videcdn.net` целиком затронет и сайт, и API одновременно. Митигируется на боевом этапе разнесением на разные SLD и несколькими API-доменами (см. AC-4.7). `[УТВ]`
- **AC-1.5.** Управление DNS — через Cloudflare API (динамическое обновление записей при ротации IP узлов). `[УТВ]`
- **AC-1.6.** Маркетинговый сайт (`www`) в текущем объёме **не разрабатывается** — фокус на двух веб-фронтендах (кабинет, админка). `[УТВ]`

## AC-2. Слой control-plane (обзор компонентов)

- **AC-2.1. API-сервис** — Go + Chi (роутер) + pgx (драйвер PostgreSQL). Отдаёт клиенту параметры (regions/config/nodes), обрабатывает регистрацию/логин, содержит бизнес-логику. `[УТВ]`
- **AC-2.2. Личный кабинет** — веб-фронтенд для пользователей. Go + Chi + `html/template` + HTMX + Bootstrap. Server-side rendering, без SPA. `[УТВ]`
- **AC-2.3. Админ-панель** — веб-фронтенд для владельца. Тот же стек. **Отдельный бинарник/сервис** (изоляция поверхности безопасности). Доступ по IP-allowlist. `[УТВ]`
- **AC-2.4. PostgreSQL** — единая БД для всех сервисов control-plane. `[УТВ]`
- **AC-2.5. Redis — НЕ используется на старте.** Потребности (сессии, кэш `/config`/`/regions`, rate-limit) закрываются PostgreSQL + in-memory кэшем внутри Go-процесса. Redis — в бэклог, оправдан при горизонтальном масштабировании API (несколько инстансов за балансировщиком, где нужен общий кэш/сессии/rate-limit). `[УТВ]`
- **AC-2.6. Деплой** — Docker + Docker Compose. Сервисы: `api`, `cabinet`, `admin`, `postgres`. `[УТВ]`
- **AC-2.7. Размещение БД:** на этапе теста — на том же KZ-сервере, что и edge. На боевом этапе — отдельный control-plane хост (БД пользователей/биллинг физически отделены от расходных edge-узлов). Схема не должна завязываться на локальные особенности edge-сервера, чтобы переезд был простым. `[УТВ]`

## AC-3. Слой data-plane (edge-узлы и протоколы)

- **AC-3.1.** Каждый edge-узел поднимает одновременно: VLESS Reality (Xray-core, под управлением 3X-UI, TCP/443) и Hysteria2 (отдельный `hysteria-server` через systemd, UDP/8443). `[УТВ]`
- **AC-3.2.** Провижининг пользователей на edge — через абстракцию `EdgeProvisioner` (см. AC-6). Реализация для VLESS — 3X-UI API; для Hysteria2 — native auth (статический список паролей на старте). `[УТВ]`
- **AC-3.3.** Учёт трафика на edge: 3X-UI ведёт статистику по каждому client'у (VLESS). Данные периодически стягиваются в БД (см. поток телеметрии AC-6.5). `[УТВ]`
- **AC-3.4.** На старте пул/балансировщик не реализуется (1 узел). Модель данных готова к пулу. `[УТВ]`
- **AC-3.5. Известные протокольные ограничения** (из DPI-находок, влияют на клиентскую логику AC-7):
  - VLESS Reality блокируется у конкретного провайдера/локации (Ростелеком, дом) — тишина после ClientHello (`ref E2/M1`). Формулировки в проекте — «блокируется у провайдера/в локации», не «в РФ» (`ref M2`).
  - Hysteria2 в некоторых сетях (корп.) рвётся ровно через 30с (`maxIdleTimeout`) — резерв не универсален (`ref L2`). Кандидат-митигатор: port hopping (`ref L6`, бэклог).
  - Вывод: два протокола не покрывают все сети; третий транспорт (TUIC/obfuscated Shadowsocks) — бэклог. `[УТВ]`

## AC-4. Модель запросов клиент → API (bootstrap-поток)

Три логически независимых эндпоинта. Клиент вправе запрашивать их пачкой/параллельно, когда удобно.

- **AC-4.1. Запрос 0 — `GET /api/v1/regions`.** Список доступных регионов (для отрисовки в GUI). Кэшируется локально с TTL; при повторном старте рисуется из кэша, запрос 0 обновляет GUI в фоне только при изменении. `[УТВ]`
- **AC-4.2. Запрос 1 — `GET /api/v1/config`.** Глобальные параметры сервиса. Длинный TTL. Содержит (см. AC-5): health-check URL, интервалы, таймауты, параметры self-check, backoff, `config_ttl`. `[УТВ]`
- **AC-4.3. Запрос 2 — `GET /api/v1/nodes?region=<code>`.** Список узлов региона с параметрами подключения и **индивидуальными учётными данными пользователя** (VLESS-UUID/Hysteria2-пароль для устройства). Короткий TTL. Требует валидной сессии; бизнес-логика проверяет право доступа при каждой выдаче. `[УТВ]`

**Сценарии вызова (UX-триггеры):** `[УТВ]`
- **AC-4.4. Первый запуск:** старт → запрос 0 → пользователь выбирает регион → запрос 1 + сохранение региона локально → (запрос 2 при коннекте).
- **AC-4.5. Повторный запуск, регион известен, без автоконнекта:** старт → запрос 0 (отрисовка GUI, можно из кэша) ∥ запрос 1 → затем запрос 2 при коннекте.
- **AC-4.6. Повторный запуск, регион известен + автоконнект:** старт → запрос 0 **пропускается** → запрос 1 → запрос 2 → коннект.

- **AC-4.7. Редундантность API (боевой этап).** Несколько независимых API-доменов на разных TLD/регистраторах; в клиент зашит fallback-список доменов; при отказе основного — перебор следующего (по образцу `getUrlFromStorage/fetchActiveUrl` Browsec). На этапе теста — один домен. `[ДЕФОЛТ]`
- **AC-4.8. Подпись ответа config/nodes** (Ed25519, публичный ключ зашит в клиент) — защита от подмены параметров подключения/health-check URL. **Бэклог** (security-hardening перед боевым запуском). `[УТВ]`

## AC-5. Config-JSON (структура ответа `/config`)

Логически две части; на уровне API — эндпоинты AC-4.2 (глобальное) и AC-4.3 (узлы) раздельны. Ниже — глобальная часть (`/config`).

Поля глобального конфига `[ДЕФОЛТ, значения уточняются на тестах]`:
- `config_ttl_ms` — срок валидности конфига у клиента.
- `health_check_url` — URL проверки живости туннеля. Дефолт, зашитый в клиент: `https://www.google.com/generate_204`. Актуальное значение приходит с сервера (переопределяет дефолт).
- `health_check_interval_ms` — интервал проверки (ориентир 10000, по образцу Browsec `observatory_probe_interval_ms`).
- `connect_timeout_ms`, `max_connect_attempts` — параметры попытки подключения.
- `self_check` — объект:
  - `ip_echo_urls` — список сервисов эхо-IP (`icanhazip.com`, `api.ipify.org`, `ifconfig.me/ip`, `checkip.amazonaws.com`).
  - `match_mode` — **`asn` / `range`** (НЕ `exact`). Урок `ref F1/F2`: побайтовая сверка даёт ложные `ip_mismatch`.
- `reconnect_backoff` — параметры лестницы восстановления (базовый интервал, множитель, максимум).

> Health-check URL и self-check-параметры **приходят с сервера** и потому входят в объём подписи ответа (AC-4.8).

## AC-6. Бизнес-логика (service layer) и управление доступом

### AC-6.1. Слоистая архитектура

```
HTTP-хендлеры (Chi)
      │
Бизнес-логика / service layer   ← правила: право доступа, генерация учёток,
      │                            провижининг, отзыв, учёт трафика
Репозиторий (pgx) ⇄ PostgreSQL
      │
EdgeProvisioner (адаптеры) → 3X-UI API (VLESS) / Hysteria2 auth (Hysteria2)
```
Хендлеры не знают SQL; репозиторий не знает правил; бизнес-логика не знает HTTP. `[УТВ]`

### AC-6.2. Разделение идентификаторов (критично)

- **`user_id`** — внутренний PK пользователя (UUID). **Никогда не покидает control-plane**, не уходит на edge, не в трафике.
- **Учётные данные туннеля** (VLESS-UUID, Hysteria2-пароль) — отдельные сущности в таблице `user_credentials`, привязаны к `user_id`. Уходят на edge и клиенту. Ротируемы/отзываемы независимо от аккаунта.
- Причины разделения: безопасность (компрометация edge не раскрывает внутренние ID), ротация (перевыпуск при утечке без смены аккаунта), множественность (**UUID на устройство**). `[УТВ]`

### AC-6.3. Генерация учётных данных

- При регистрации генерируются **сразу оба**: VLESS-UUID (основной) и Hysteria2-пароль (резерв). `[УТВ]`
- Модель — **«credential на устройство»**: у пользователя может быть несколько активных учёток (по устройствам), отзыв на одном устройстве не трогает остальные. `[УТВ]`

### AC-6.4. Поток провижининга и отзыва

1. Регистрация → запись в `users`, генерация учёток → `user_credentials`. Присвоение триала (см. AC-6.6).
2. Пуш учёток на активные edge-узлы региона через `EdgeProvisioner` (VLESS → 3X-UI API; Hysteria2 → native auth).
3. Запрос 2: бизнес-логика по сессии определяет пользователя, проверяет право доступа (статус/лимит), формирует JSON с **его** учётками для узлов. При закрытом доступе — пустой список/ошибка с причиной.
4. Подключение: клиент идёт на edge напрямую; edge аутентифицирует локально.
5. **Отзыв доступа** = (а) прекратить выдачу узлов в запросе 2 **И** (б) удалить учётку с edge через `EdgeProvisioner`. Только (а) недостаточно — со старым сохранённым JSON доступ сохранится, пока учётка жива на edge. `[УТВ]`

### AC-6.5. Учёт трафика (телеметрия)

- Отдельный поток телеметрии: периодический опрос статистики трафика по client'у из 3X-UI → запись в БД (см. `traffic_usage`, AC-DB). `[УТВ]`
- Лимит (в т.ч. 100 МБ триала) считается **на стороне бизнес-логики** (не делегируется 3X-UI) — так как всё равно нужен единый расчёт при пуше на десятки узлов. `[УТВ]`

### AC-6.6. Статусы пользователя

- Поле `users.status`: `trial` / `active` / `expired` / `blocked` (`VARCHAR` + CHECK). `[УТВ]`
- Триал: 5 дней / 100 МБ `[ДЕФОЛТ, уточняется]`.
- **Переходы между статусами** (trial→active→expired, blocked и т.д.) — `[ОТЛОЖЕНО→чат Бэкенд/Биллинг]`, т.к. завязаны на оплату. Здесь фиксируется только перечень статусов как поле БД.

### AC-6.7. Reconciler (желаемое vs фактическое состояние)

- БД хранит желаемое состояние (кто должен быть на каких узлах). Фоновый процесс сверяет фактическое состояние каждого edge с желаемым и досинхронизирует (устойчивость к недоступности узла в момент пуша). Паттерн desired-vs-actual (аналогия с Kubernetes). `[УТВ]`
- На старте (1 узел) — прямой пуш. Reconciler — в модель сейчас, в реализацию при втором узле. `[УТВ]`

### AC-6.8. EdgeProvisioner — интерфейс-абстракция

- Методы (ориентир): `AddUser`, `RemoveUser`, `RotateCredential`, `GetTraffic`.
- Реализации: `ThreeXUIProvisioner` (VLESS через 3X-UI API), `Hysteria2Provisioner` (native auth, статический список паролей на старте; HTTP external auth — при >1000 пользователей).
- Изоляция: смена панели/переход на прямое управление Xray = замена адаптера, не бизнес-логики. `[УТВ]`
- **3X-UI как производное состояние:** панель — для просмотра и аварийного вмешательства, НЕ для рутинного управления (иначе рассинхрон с БД-источником истины). `[УТВ]`

### AC-6.9. Аутентификация к API

- email + пароль; `password_hash` — argon2id `[ДЕФОЛТ]`. OAuth (GitHub как естественный для dev-аудитории) — `[ОТЛОЖЕНО→чат Бэкенд/Биллинг]`.
- Сессии — **serverside, токен в БД** (`sessions`), не JWT (мгновенный отзыв, гибче). `[УТВ]`

## AC-DB. Схема базы данных (PostgreSQL)

### AC-DB-0. Общие соглашения

- **PK:** `UUID` (генерация `gen_random_uuid()`), на всех публично-адресуемых сущностях. `[УТВ]`
- **Временные метки:** `created_at TIMESTAMPTZ NOT NULL DEFAULT now()`, `updated_at TIMESTAMPTZ NOT NULL DEFAULT now()` — во всех сущностях **кроме событийных/журнальных** таблиц (там только момент записи). Всё в UTC (`TIMESTAMPTZ`). `[УТВ]`
- **`updated_at`** обновляется триггером `BEFORE UPDATE` (не полагаемся на приложение). `[УТВ]`
- **Soft delete:** поле `deleted_at TIMESTAMPTZ NULL` в `users` и `nodes` (одинаково). Записи не удаляются физически при штатном удалении; фильтрация `WHERE deleted_at IS NULL`. `[УТВ]`
  - **Оговорка (privacy):** физическое удаление персональных данных при удалении аккаунта пользователя — `[ОТЛОЖЕНО→чат Юрлицо/Комплаенс]` (политика хранения/no-logs). Soft delete — дефолт механики, но право на физическое стирание PII нужно учесть отдельно.
- **Enum-поля:** `VARCHAR` + `CHECK`-constraint (не нативный `ENUM`) — проще миграции/добавление значений. `[УТВ]`
- **Секреты туннеля** (`user_credentials.credential`) — **шифруются на уровне приложения** (encryption at rest отдельным ключом), т.к. нужны в открытом виде для подключения (хэшировать нельзя). `password_hash` — именно хэш. `[УТВ]`

## AC-7. Клиентская логика адаптивности (Flutter)

- **AC-7.1. Справочник «сеть → протокол».** Локальный на клиенте: `network_id → protocol`. `network_id` — стабильный идентификатор сети (хэш SSID/MAC шлюза — уточнить, что доступно на каждой ОС). При подключении первым пробуется протокол, сработавший на этой сети в прошлый раз. Обоснование: дома (Ростелеком) — Hysteria2, на работе — VLESS (`ref M1/L4/L2`). `[УТВ]`
- **AC-7.2. Приоритет по умолчанию** (незнакомая сеть): VLESS Reality → Hysteria2 (AC-0.3). `[УТВ]`
- **AC-7.3. Порядок попыток.** Скорость первого подключения не критична → допустим последовательный перебор (меньше одновременных исходящих ClientHello, меньше собственный сетевой «шум»). `[УТВ]`
- **AC-7.4. Health-check поверх туннеля.** Периодическая проба (URL и интервал из `/config`, дефолт `google.com/generate_204` / ~10с) — раннее обнаружение blackhole-паттерна (тишина без RST, `ref E2`) и деградации Hysteria2 **до** 30-секундного `maxIdleTimeout` (`ref L1/L2`). `[УТВ]`
- **AC-7.5. Лестница восстановления:** `[УТВ]`
  1. быстрый повтор на том же узле/протоколе (разовый сбой);
  2. следующий узел того же протокола (когда узлов > 1);
  3. **смена протокола** (Hysteria2 ↔ VLESS) — ядро адаптивности;
  4. экспоненциальный backoff на уровне цепочки (частый повтор — сам по себе детектируемый паттерн).
- **AC-7.6. Self-check exit-IP.** Модуль внутри Flutter-клиента: после поднятия туннеля запрашивает внешний IP через `ip_echo_urls`, сверяет с ожидаемым IP узла **по ASN/диапазону, не побайтово** (`ref F1/F2`). На Linux предварительно проверять/сбрасывать `gsettings org.gnome.system.proxy` (`ref F3`). Результат репортится в `node_health` (`source=client_report`) — это клиентская телеметрия блокировок. `[УТВ]`
- **AC-7.7. Хранение выбора региона** — локально на клиенте (кэш с TTL). Синхронизация между устройствами (через `users.preferred_region_id`) — на будущее. `[УТВ]`

## AC-8. Бэклог (зафиксированные отложенные решения)

- **AC-8.1.** Подпись config/nodes-JSON (Ed25519) — security-hardening перед боевым (AC-4.8).
- **AC-8.2.** Третий транспорт (TUIC / obfuscated Shadowsocks) — адаптивность не должна упираться в два протокола (AC-3.5).
- **AC-8.3.** Port hopping для Hysteria2 (`hopInterval`) против обрыва 30с (`ref L2/L6`).
- **AC-8.4.** Сплит-туннинг по GeoIP для РФ-доменов (`ref H1`) — снижение объёма «подозрительного» трафика + UX.
- **AC-8.5.** Hysteria2 HTTP external auth — переход со статического списка паролей при >1000 пользователей (AC-6.8).
- **AC-8.6.** Reconciler (мультиузловая синхронизация) — реализация при втором узле (AC-6.7).
- **AC-8.7.** Redis — при горизонтальном масштабировании API (AC-2.5).
- **AC-8.8.** Редундантность API-доменов + fallback-список в клиенте (AC-4.7); разнесение сайта/API на разные SLD (AC-1.4).
- **AC-8.9.** FakeDNS для полного покрытия OS-трафика через TUN (`ref G3`) — если/когда потребуется unified TUN.
- **AC-8.10.** Физическое стирание PII при удалении аккаунта (privacy) — политика в чате Юрлицо/Комплаенс (AC-DB-0).
