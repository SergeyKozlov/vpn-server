## Docker / ufw — обход фаервола, живучесть правил

Источник: `ai_from_local/Hardening_Deploy_Report.md` (2026-07-22).

- **Docker правит iptables/NAT напрямую и может обойти ufw.** Опубликованный порт контейнера (`ports:` в compose) становится доступен через DNAT-правила в цепочке `DOCKER`/`DOCKER-USER`, которые Docker создаёт сам, независимо от того, что говорит `ufw status`. Проверять реальную доступность порта снаружи нужно **живой попыткой подключения**, а не чтением `ufw status`. Это не гипотетический риск — именно так была устроена путаница, описанная ниже.
- **Найдено при хардненинге:** пакет `ufw` в какой-то момент был вручную удалён (`dpkg` показывал `rc` — конфиги остались, бинаря нет), но правила в iptables продолжали жить через `iptables-persistent`/`netfilter-persistent`, отдельно от `ufw`. `INPUT` policy была `ACCEPT` — то есть отсутствие `ufw`-пакета не значило "всё закрыто", это значило "правила есть, но их состоянием никто не управляет через ufw". Переустановка `ufw` (в Debian это `Breaks: iptables-persistent, netfilter-persistent`) сносит эти два пакета как побочный эффект.
- **Вывод для будущих задач:** после любого действия с фаерволом/портами — проверять not только `ufw status`, но и `iptables -L DOCKER-USER -n -v`, `iptables -t nat -L DOCKER -n -v`, и (главное) реальную попытку подключения снаружи. Это правило уже применено в спеке `phase-2-wireguard/postgres-over-tunnel` (`~/vpn-iac`).

## `127.0.0.1` внутри bridge-контейнера ≠ `127.0.0.1` хоста

Источник: `ai_from_local/VLESS_3xUI_Connectivity_Fix_Report.md` (2026-07-22).

- Каждый контейнер на Docker bridge-сети (`network_mode` по умолчанию, не `host`) получает свой собственный network namespace и свой loopback. `127.0.0.1` внутри такого контейнера — это loopback самого контейнера, а не хоста и не другого контейнера/процесса на хосте.
- **Конкретный инцидент:** `api` (bridge-сеть `vpn-net`) не мог достучаться до панели 3x-ui (`network_mode: host`, слушает `*:2053`), потому что `XUI_BASE_URL` был `http://127.0.0.1:2053` — внутри `api` это адрес самого `api`, где ничего не слушает 2053. Ошибка: `dial tcp 127.0.0.1:2053: connect: connection refused`.
- **Фикс:** `extra_hosts: ["host.docker.internal:host-gateway"]` в сервисе `api` + `XUI_BASE_URL=http://host.docker.internal:2053`. `host.docker.internal` через `host-gateway` резолвится в IP шлюза bridge-сети (`172.18.0.1` в этой конфигурации), откуда виден хост и любой `network_mode: host` контейнер/процесс на нём.
- **Где ещё это всплывёт:** любой bridge-networked сервис, которому нужно достучаться до хоста, другого `network_mode: host` сервиса, или (актуально для WireGuard) до IP туннельного интерфейса `wg0`, поднятого на самом хосте — не может использовать `127.0.0.1` для этого. Нужен либо `host.docker.internal`/`host-gateway`, либо явный IP хоста/интерфейса, либо правильное сетевое подключение контейнера.

## Топология Hysteria2 — реальный процесс живёт вне Docker

Источник: `ai_from_local/Hysteria2_Reload_Diagnostic_Report.md` (2026-07-22).

- **`docker compose ps -a` вводит в заблуждение**: compose-сервис `hysteria2` (`tobyxdd/hysteria:latest`) — `Exited`, не обслуживает трафик. Реальный процесс на UDP `:8443` — голый бинарь `/usr/local/bin/hysteria`, запущенный systemd-юнитом `hysteria-server.service` (не `hysteria2` — это красная селёдка при поиске), полностью вне Docker.
- **Два разных config.yaml с одинаковым именем:** файл, который переменная `HYSTERIA_CONFIG_PATH` (`/root/vpn/hysteria/config.yaml`) указывает читать/писать приложению `api`, и файл, который реально читает живой процесс (`/etc/hysteria/config.yaml`) — **разные файлы**, разные inode, разный контент (в живом есть секции `masquerade`/`bandwidth`, в другом нет). Запись по `HYSTERIA_CONFIG_PATH` никогда не долетит до живого процесса.
- **Reload недостижим из контейнера `api` в принципе** без эскалации привилегий: нет общего PID namespace, нет `docker.sock`, нет systemd control socket, всё это явно запрещено рамками задачи, где это обнаружено. `api`-контейнер (`distroless`, нет `sh`) физически не может выполнить `systemctl restart` или `kill -HUP` на хостовый процесс.
- **Вывод:** `POST /clients` в приложении сейчас нефункционален для Hysteria2 (и отдельно — для VLESS, см. пункт про `127.0.0.1` выше — обе проблемы блокируют один и тот же код-путь). Рекомендованный (но не выполненный) фикс — перенести Hysteria2 под compose на ту же bridge-сеть, что и `api`, что заодно устранит и "два конфига", и reload-проблему. Это в бэклоге, не в этой фазе IaC.
- **Релевантно для IaC:** любая будущая Ansible-роль, которая трогает Hysteria2 на bastion, должна ориентироваться на реальный systemd-юнит `hysteria-server.service` и реальный путь `/etc/hysteria/config.yaml` — не на compose-сервис `hysteria2`, который сейчас не обслуживает трафик.

## Протоколы / API — сводка для контекста (не DPI-специфика)

Источник: `ai_from_local/compass_artifact.md` (2026-07-19) — технический референс по 3X-UI API, Hysteria2 auth, Caddy DNS-01, задокументированный для Фазы 2 приложения (не относится к сетевой IaC-фазе, но фиксируется здесь как единственное место в репозитории, где это было исследовано).

- **3X-UI**: весь API живёт под префиксом `{webBasePath}`, включая `/login`. Сессионная кука называется буквально `3x-ui`. Для automation предпочтителен Bearer API Token (Settings → Security), не cookie-логин.
- **Hysteria2**: только `auth.type: userpass` даёт нативные per-user credentials без внешнего сервиса; `type: password` — общий секрет на всех.
- **Caddy**: порт 8443 занят, поэтому TLS получают через DNS-01 (Cloudflare), а не HTTP-01/TLS-ALPN-01 — не нужен внешний доступ к 80/443. Требует кастомную сборку с `caddy-dns/cloudflare`.
- Полный референс (payload-примеры, коды полей, версии) — см. сам файл, если понадобится при работе над Фазой 3+ (приложение на host_b).
